# 回归测试

主流程步骤 9。改完代码后执行。**先按本地/线上分流**，切勿对线上目标改完就直接回归。

## 情况 A：本地 URL → 可直接回归

确认改动已生效（dev 热更新，或 `build` + `preview`/`start`——**回归须用生产构建**），重新执行 [phase-a-audit.md](phase-a-audit.md) 分析步骤 1–3（同一 MCP/CLI 模式与 device），对比前后指标。

## 情况 B：线上 URL → 先提醒部署

改完代码后**停在这里**：

> 已完成代码修改（列出文件/项）。改动尚未上线，**直接回归只会测到旧代码**。请先部署/发布，完成后再回归对比。

用户回复「已部署」后，**先做部署校验**：

- `evaluate_script` 查 DOM 属性
- `list_network_requests` 查请求次数
- `curl -sI` 查响应头

确认至少一项改动特征（`lang`、`fetchpriority`、缓存头、请求次数下降等）**已在线上生效**；未生效则中止回归。

校验通过后重新执行分析步骤 1–3，采集指标对比。

## 共用规则

- 与首次审计**同一模式（MCP/CLI）与同一 device**。
- 先验证修复特征生效，再采集完整指标。
- 回归通过才关闭任务。
- 收尾**最多一句**：CrUX 现场数据需时间反映，不展开科普。
